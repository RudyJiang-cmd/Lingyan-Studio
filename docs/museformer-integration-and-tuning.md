# Lingyan Studio：Museformer 接入与音乐性调试复盘

本文记录当前版本从纯前端原型，演进到“前端网站 + 后端模型服务器 + Museformer 推理 + 四声部回放”的完整过程，供后续维护、答辩、复现和继续调音时使用。

- 线上前端：`http://119.45.247.15`
- GitHub：`https://github.com/RudyJiang-cmd/Lingyan-Studio`
- AI 后端：`http://43.129.229.99:8000/generate`
- 当前状态：
  - 前端已经真实请求 Museformer 后端
  - 后端已经以 `systemd` 方式稳定运行
  - Museformer 推理链路已经打通
  - “播放”会同时播放玩家主旋律和 AI 生成声部
  - 录音失败、AI 超时、后端 500 等错误提示已经明确化
  - 和声从“经常只生成一个声部”提升到“稳定补出 alto / tenor / bass 三层”

## 2. 系统结构

### 2.1 服务器分工

| 角色 | 地址 | 实际职责 |
| --- | --- | --- |
| 前端静态站点 | `119.45.247.15` | Nginx 托管 Vite 构建产物 |
| 模型后端 | `43.129.229.99` | FastAPI + Museformer 推理服务 |
| 代码仓库 | GitHub `RudyJiang-cmd/Lingyan-Studio` | 前后端代码、部署脚本、文档 |

### 2.2 请求链路

1. 用户在前端五线谱上绘制主旋律。
2. `src/App.tsx` 取出 `voice === 'user'` 的音符，组装成 `[{ step, pitch }]`。
3. 前端向 `http://43.129.229.99:8000/generate` 发起 `POST` 请求。
4. `server/museformer_api.py` 将旋律转成 MIDI，再编码成 Museformer 使用的 `REMIGEN` token。
5. FastAPI 调用 Museformer 的 `fairseq` 推理。
6. 模型输出 token 被解析为每个 step 的候选音高。
7. 后处理逻辑把候选音高重新分配为 `alto / tenor / bass`。
8. 前端收到 `generated_notes` 后渲染到谱面，并在播放时与主旋律一起回放。

## 3. 接入前的真实情况

最初线上实际运行的是纯静态前端：

- `119.45.247.15` 上只有 Nginx
- 对外生效的 JS 包仍然是本地规则生成和声
- 页面文案甚至还保留“未来将接入 AI 音乐大模型”
- 也就是说：
  - 本地源码里已经写了 AI 接口，不等于线上真的在用
  - 买了新 GPU 服务器，不等于前端已经切过去

因此排查被拆成两条独立链路：

1. 前端部署链路：线上是否真的在请求 `43.129.229.99:8000/generate`
2. 模型服务链路：`43.129.229.99` 上的 Museformer 是否真的能返回结果

## 4. 前端接入过程

### 4.1 真正切到 AI 后端

核心文件：`src/App.tsx`

关键改动：

- 把“生成和声”按钮接到真实 `fetch`
- 请求目标改为 `http://43.129.229.99:8000/generate`
- 请求失败时不再静默退回本地假生成
- 明确处理：
  - 超时
  - 后端 500
  - 网络错误
  - 空旋律

### 4.2 明确错误提示

为了避免只看到“AI 连接失败”这种无信息提示，前端增加了更细的失败分支：

- `AI 服务器响应超时（15 秒）`
- 后端返回 500 时展示 `detail`
- 录音失败会明确区分：
  - 页面不是安全上下文
  - 麦克风权限被拒
  - 没有麦克风设备
  - 麦克风被占用

### 4.3 多声部回放

之前“播放”只播放玩家自己输入的主旋律。当前已改成播放所有音符：

- `user`：`triangle`
- `alto`：`sine`
- `tenor`：`square`
- `bass`：`sawtooth`

同时对 `tenor / bass` 做了八度下移，让浏览器回放时四个声部更容易分辨。

### 4.4 录音功能为什么会“失效”

不是业务代码崩了，而是浏览器安全策略：

- 当前站点是 `http://119.45.247.15`
- `getUserMedia` 在很多浏览器中要求 `HTTPS` 或 `localhost`
- 因此线上 IP 直连场景下，录音功能会被浏览器拒绝

这个问题现在已经会在前端直接提示，但要真正修复，仍然需要把线上站点切到 HTTPS。

## 5. 部署链路修复

核心文件：`deploy_pem.exp`

实际踩到的问题是一个“假部署”：

- 压缩包被解到了 `/home/ubuntu/dist/dist/`
- 但脚本复制的是 `/home/ubuntu/dist/*`
- 结果就是：新文件上传了，但 `index.html` 仍然可能指向旧 JS

修复后策略：

- 每次部署使用独立时间戳目录
- 在远端解压到单独的 `deploy-YYYYMMDD-HHMMSS`
- 再从正确的 `dist/*` 复制到 `/var/www/html/`

这一步解决了“本地源码已经改了，但线上还在跑旧包”的核心假象。

## 6. Museformer 后端接入过程

核心文件：`server/museformer_api.py`

### 6.1 服务化

最初的后端只是挂在 SSH 会话里的 Python 进程，不稳定，也不便于诊断。

后续改成：

- `systemd` 服务：`museformer-api.service`
- 统一由 `systemctl` 拉起
- 可复现重启
- 可检查状态
- 可通过日志定位模型加载与推理卡点

### 6.2 Museformer 的实际工作方式

当前服务不是把前端 JSON 直接喂给模型，而是走了这条链：

1. 前端旋律 JSON
2. 转成 `miditoolkit.MidiFile`
3. 使用 `MidiEncoder("REMIGEN")` 编成 token
4. 用 `fairseq` 字典编码为模型输入
5. 调 `task.inference_step(...)`
6. 把生成的 token 解析回每个 step 的候选音高
7. 再做和声后处理

这一步非常关键，因为早期的一个核心误区是：

- 直接手写 `Bar_None / Position_x / Pitch_y` 一类 token，并不等于 Museformer 真正吃得下去
- 只有编码方式与词表对齐，模型才会输出可用结果

## 7. 调试过程中踩到的关键问题

### 7.1 线上前端其实没切过去

现象：

- 本地代码看起来已经在请求 AI
- 线上点击却仍然秒出本地和声

原因：

- `119.45.247.15` 对外服务的还是旧静态包

修复：

- 重建前端
- 修好部署脚本
- 重新上线

### 7.2 后端服务能起，但推理卡死

现象：

- `8000` 端口开着
- `/generate` 长时间超时

后续确认的问题包括：

- `torch_extensions` 的 `block_fill_cuda` 残留缓存和锁问题
- 服务进程挂在 SSH 会话下，不稳定

修复：

- 清理扩展缓存问题
- 切换到 `systemd`

### 7.3 `prefix_tokens` 维度错误

现象：

- 推理时报 `Dimension out of range`

修复：

- 从 `prefix_tokens=input_ids`
- 改为 `prefix_tokens=input_ids.unsqueeze(0)`

### 7.4 token 编码与解码不匹配

现象：

- 请求返回空数组或无意义结果

原因：

- 输入 token 格式与 Museformer 实际使用的 `REMIGEN` 不一致

修复：

- 使用真实 MIDI 编码器
- 用模型字典统一编码和解码

### 7.5 “只生成一个声部”或“总撞主旋律”

这是当前最重要的音乐性问题，也是后处理重点。

## 8. 音乐性调试是怎么做的

Museformer 原始输出不是可以直接展示给用户的完整 SATB 和声，它更像是“候选音高材料”。因此必须做后处理。

### 8.1 第一阶段：先保证能稳定补满三层

做法：

- 按 step 收集候选音高
- 每个 step 强制尝试补 `alto / tenor / bass`
- 没候选时允许从允许音集合中兜底

结果：

- 从“经常只返回一个音”提升到“稳定生成三层”

### 8.2 第二阶段：避免与主旋律同音

做法：

- `used_pitches` 中先占住 `user_pitch`
- 任何生成声部不允许与主旋律同 pitch
- 同一步各声部之间也尽量不重复

结果：

- 明显减少“和声压在主旋律上”的情况

### 8.3 第三阶段：加入声部目标区间与平滑度

当前后处理会综合考虑：

- 每个声部相对主旋律的目标区间
- 前一步同声部的音高，减少大跳
- `alto / tenor / bass` 的可视音域范围
- 优先选择音域内候选
- 如果候选池不理想，优先使用音域内兜底，再考虑极端值

这一步的目标是：

- 不只要“有音”
- 还要避免 `alto` 突然跑到最极端位置
- 让 `tenor / bass` 的行为更像声部，而不是随机填空

### 8.4 当前效果

相较于早期版本，现在已经有两个明确提升：

1. 三个生成声部更稳定
2. 极端跳到 `pitch 0` 一类不自然情况已被压下去

但仍然还有明显改进空间：

- `bass` 仍然偏保守，容易重复同一落点
- `alto / tenor` 还不够像真正的“和声进行”
- 终止感、张力释放、和弦功能都还没有被系统控制

## 13. 2026-04-30 前端交互与乐谱整理更新

本轮把乐谱输入重新收敛到稳定、可理解的四小节编辑体验：玩家仍然按最小四分音符颗粒放置音高，系统在播放和生成和声前自动整理时值，让乐谱看起来更接近成立的四四拍旋律。

### 13.1 输入模型

- 编辑器固定为 16 个四分音符格，构成 4 个四四拍小节。
- 玩家不需要手动选择休止符、八分音符或更短时值，避免输入阶段过度复杂。
- 鼠标绘制逻辑保持轻量：点击或拖动只负责落下音高点。

### 13.2 自动时值整理

- 每个音会自动延长到同小节内的下一个音之前。
- 如果一个小节里只有一个音，这个音会延长成覆盖整小节的全音符效果。
- 自动整理只发生在播放与 AI 和声生成前，不干扰玩家编辑时的直觉输入。
- 前端会把整理后的 `duration` 一并发送给后端，Museformer 路径也能保留音符时值。

### 13.3 视觉更新

- 乐谱宽度会根据浏览器宽度动态压缩，保证四个小节和终止线完整可见。
- 小节线、终止线、谱号和拍号重新对齐到同一套谱面坐标。
- 音符外观改为 duration-aware：四分音符是实心符头带符干，二分/附点二分效果为空心符头带符干，全小节延长效果为空心符头且不显示符干。
